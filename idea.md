# Система проведения турниров по швейцарской системе (Chess Swiss)
**Единый документ требований + план итераций**  
Версия: **0.2** (ориентация на быстрые релизы / Scrum)  
Дата: **2026‑03‑15**

---

## 1. Цели и границы

### 1.1. Цели продукта
- Автоматизировать проведение шахматных турниров по швейцарской системе: регистрация участников, жеребьёвка, ввод результатов, расчёт таблицы и тай-брейков.
- Предоставить **админку** для главного судьи (и при необходимости помощников).
- Предоставить **публичный фронт** для публикации расписания (пар), результатов, таблицы (актуальной и “после каждого тура”).
- Предоставить доступ множеству аккаунтов (SaaS) 
- Поддержать **множество турниров** (multi‑tournament) внутри одного аккаунта.

### 1.2. Принципы
- **Быстрые поставки**: функциональность поставляется итерациями (Scrum), вертикальными срезами, с ранним MVP.
- **Pairing engine — сердце**: развивается постепенно; на MVP допускается упрощённый алгоритм + возможность ручной корректировки судьёй.
- **История важнее красоты**: изменения пар/результатов и критичных настроек логируются.

### 1.3. Вне рамок проекта
- Командные турниры.
- Полная интеграция “из коробки” с внешними рейтингами (FIDE/национальные) через API.
- Игровая платформа (онлайн‑партии) — система фиксирует результаты, а не проводит игры.

## 2. Роли и доступ

Роли назначаются **на уровне турнира** (один пользователь может быть судьёй в турнире A и гостем в турнире B).

- **Public (Гость)**: читает публичные страницы турнира.
- **User (Пользователь)**: зарегистрирован, может создавать турниры (если разрешено политикой).
- **Organizer (Организатор)**: владелец турнира, управляет настройками, назначает роли.
- **Chief Arbiter (Главный судья)**: ведёт турнир: участники, жеребьёвка, результаты, публикации.
- **Assistant Arbiter (Помощник судьи)**: ввод результатов, помощь (опционально).
- **System Admin**: управление всей платформой (опционально).

---

## 3. Доменные сущности (модель данных на уровне требований)
- **User**: аккаунт, профиль.
- **Tournament**: параметры турнира, статус, конфигурация жеребьёвки и тай‑брейков, публичность.
- **TournamentRole**: связь User ↔ Tournament с ролью.
- **Participant**: участник турнира, рейтинг/посев, статус.
- **Round**: тур, статус, время, публикация.
- **Pairing/Match**: пара/партия на доске, цвета, результат, тип результата.
- **Standing Snapshot**: таблица “после тура N” (для истории и стабильности).
- **Audit Log**: неизменяемый журнал действий судьи/организатора.

## 4. Функциональные требования (FR)

### A. Аккаунты и доступ
- **FR‑A1** Регистрация (email/пароль) и вход/выход.
- **FR‑A2** Восстановление пароля.
- **FR‑A3** Профиль пользователя (минимум: имя/ник).
- **FR‑A4** RBAC на уровне турнира (Organizer/Chief/Assistant).
- **FR‑A5** Изоляция данных: пользователь видит/редактирует только то, на что имеет права.

### B. Управление турнирами (много турниров)
- **FR‑B1** Создание турнира (мастер): название, описание, контроль времени, число туров, система очков, публичность.
- **FR‑B2** Статусы турнира: Draft → Registration/Open → Running → Finished → Archived.
- **FR‑B3** Список “Мои турниры”, поиск/фильтры.
- **FR‑B4** Доступ по публичной ссылке (slug).

### C. Участники
- **FR‑C1** Добавление/редактирование/удаление (с ограничениями после старта).
- **FR‑C2** Импорт участников (CSV/Excel) (позже).
- **FR‑C3** Посев: сортировка по рейтингу (если задан) или ручной seed/TPN.
- **FR‑C4** Статусы: Active / Withdrawn / Disqualified.
- **FR‑C5** Карточка участника: партии по турам, очки, соперники, цвета, тай‑брейки.

### D. Жеребьёвка (Swiss Pairing Engine)
- **FR‑D1** Генерация жеребьёвки на тур N одной кнопкой.
- **FR‑D2** Поддержка минимум двух режимов (по roadmap):
  - **Базовый (MVP / клубный)**: группировка по очкам, пары внутри группы, запрет повторов “по возможности”, простая балансировка цветов, bye при нечётном числе.
  - **Расширенный (FIDE‑похожий / Dutch‑подобный)**: score groups, floaters, цветовые предпочтения и ограничения, минимизация “прыжков”, объяснимость.
- **FR‑D3** Запрет повторных встреч (обязателен; если невозможно — стратегия деградации).
- **FR‑D4** Bye: автоматическое назначение pairing‑allocated bye при нечётном числе, запрет повторного bye одному и тому же игроку (по умолчанию).
- **FR‑D5** Requested bye (опционально): судья отмечает пропуск, система начисляет 0/0.5/1 по настройке.

### E. Туры, публикация, результаты
- **FR‑E1** Создание тура (черновик) → публикация (видно на фронте).
- **FR‑E2** Ввод результатов: 1‑0 / 0‑1 / 0.5‑0.5 / bye / forfeit / tech result.
- **FR‑E3** Пересчёт таблицы после каждого результата (live) или после закрытия тура (настройка).
- **FR‑E4** Закрытие тура: запрет закрытия при незаполненных результатах (по умолчанию).
- **FR‑E5** Откат/исправление результата с логированием причины (после публикации).

### F. Таблица и тай‑брейки
- **FR‑F1** Таблица: очки, партии, победы/ничьи/поражения, дополнительные показатели.
- **FR‑F2** Порядок сортировки: очки → тай‑брейки → посев (как последний детерминирующий критерий).
- **FR‑F3** Тай‑брейки (по roadmap): Buchholz (общий/median/cut), Sonneborn‑Berger, wins, head‑to‑head, avg opponent rating.
- **FR‑F4** История: “таблица после каждого тура” (snapshot).

### G. Публичный фронт
- **FR‑G1** Публичная страница турнира: описание, статус, участники.
- **FR‑G2** Туры: пары и результаты по каждому туру.
- **FR‑G3** Таблица: актуальная + “после тура N”.
- **FR‑G4** Быстрое обновление: polling (MVP) → realtime (позже).

### H. Админка судьи
- **FR‑H1** Панель “следующий шаг” (чек‑лист турнира).
- **FR‑H2** Табличный ввод результатов.
- **FR‑H3** Возможность ручной корректировки пар (с логом и причиной).
- **FR‑H4** Audit log по турниру: пары/результаты/участники/настройки.

### I. Экспорт и интеграции (позже)
- **FR‑I1** Экспорт (CSV/PDF) пар и таблиц.
- **FR‑I2** Read‑only API для внешних виджетов/табло.
- **FR‑I3** Уведомления (email/telegram/webhook) о публикации тура/изменениях.

## 5. Нефункциональные требования (NFR)

### Производительность
- **NFR‑1** Генерация жеребьёвки:
  - до 200 участников: целевой бюджет < 1–2 сек,
  - до 1000 участников: целевой бюджет < 5–10 сек (в будущем, при оптимизациях).
- **NFR‑2** Публичные страницы: быстрый отклик и кэширование (особенно таблица и пары).

### Надёжность и целостность
- **NFR‑3** Атомарность операций: публикация тура, закрытие тура, изменение результатов.
- **NFR‑4** Audit log неизменяемый (append‑only) для критичных действий.
- **NFR‑5** Снапшоты таблицы “после тура” сохраняются и не меняются задним числом без явного действия судьи.

### Безопасность
- **NFR‑6** Хэширование паролей современными алгоритмами, rate limit на логин.
- **NFR‑7** RBAC + защита от горизонтального доступа.
- **NFR‑8** Защита веб‑приложения: CSRF/XSS, secure cookies, CORS policy для API.

### Удобство (UX)
- **NFR‑9** Админка применима на мобильном для ввода результатов.
- **NFR‑10** Минимум кликов для сценария “опубликовать тур → внести результаты → закрыть тур”.
- **NFR‑11** Понятные ошибки жеребьёвки: конфликт повторов/цветов/настроек.

### Сопровождение и качество
- **NFR‑12** Тесты: unit для pairing engine, интеграционные сценарии турнира.
- **NFR‑13** Наблюдаемость: логи, метрики времени жеребьёвки, мониторинг ошибок.
- **NFR‑14** Версионирование правил жеребьёвки в турнире (чтобы апдейты не ломали старые турниры).

### Локализация
- **NFR‑15** EN (обязательно), RU (желательно), корректные форматы даты/времени.

## 6. Технологические требования (TR) для быстрых релизов

### 6.1 Архитектура
- **TR‑1** Вертикальные срезы: UI + API/сервисный слой, чтобы быстро выпускать фичи.
- **TR‑2** Pairing engine — отдельный модуль с тестами и версионированием.
- **TR‑3** Multi‑tournament “из коробки”: все сущности привязаны к tournament_id и фильтруются по правам.

### 6.2 Рекомендуемый стек для MVP (максимально быстрый старт)
- **Backend**: Python 3.12+.
- **Framework**: Django + Django Admin + DRF.
- **DB**: PostgreSQL.
- **Auth**: сессии/secure cookies (админка).
- **Public UI**: Django templates (на MVP), далее можно заменить/добавить Next.js.
- **Deploy**: Docker + CI (тесты/линтер/сборка).

### 6.3 Улучшения “по мере роста”
- **TR‑4** Redis: кэш/блокировки/ускорение публички.
- **TR‑5** Очередь задач (Celery): импорты/экспорты/уведомления.
- **TR‑6** Observability: Sentry + метрики (Prometheus/Grafana или аналог).

### 6.4 Тестирование/качество
- **TR‑7** Unit‑тесты pairing engine (детерминированность).
- **TR‑8** Интеграционные тесты: “тур → результаты → тур → таблица”.
- **TR‑9** Линт/формат в CI (ruff/black и т.п.).

## 7. Итерации разработки (Scrum Roadmap)

### Итерация 1 — MVP “можно провести турнир”
**Ценность:** пользователь регистрируется и реально проводит турнир, публика видит пары/результаты/таблицу.

**Must‑have:**
- Регистрация/логин.
- Создание турнира (минимальные параметры).
- Админка: участники (ручной ввод), посев, генерация тура, публикация, ввод результатов, закрытие тура, следующий тур.
- Публичка: туры (пары/результаты), таблица актуальная + “после тура N”.
- Снапшот таблицы после закрытия тура.

**Pairing engine (MVP):**
- Тур 1: верхняя половина vs нижняя половина по посеву.
- Тур 2+: группировка по очкам, пары внутри группы; при нечётности — один “плавает вниз”.
- Избегать повторов пар.
- Bye при нечётном числе (по настройке очков).

---

### Итерация 2 — удобство судьи + базовые тай‑брейки
- В черновике расписания очередного тура отображать и результатах тура должно отображаться, у кого bye
- Тай‑брейки минимум: Buchholz + wins + head‑to‑head (настраиваемый порядок).
- Коэффициент Бухгольца должен быть виден в тернирной таблице

---

### Итерация 3 — Swiss Engine 2.0 (FIDE‑похоже)
- Улучшенные floaters и минимизация прыжков между score‑группами.
- Улучшенная балансировка цветов (избегать 3 одинаковых подряд, учитывать предпочтение).
- “Объяснение жеребьёвки” (score group, floater, конфликт повторов).

---

## Testing. Definition of Done для MVP (итерация 1)
- Можно создать турнир, добавить участников, провести минимум 2 тура.
- Публичная ссылка отображает пары/результаты/таблицу корректно.
- Данные турниров изолированы по пользователям и ролям.
- Жеребьёвка детерминированна.
- Есть минимальные тесты pairing engine и основной сценарий турнира.

## Управляемые компромиссы

1. Стек и деплой
- Запускаем в VPS, используем github + github actions
- Для быстрого старта Django + DRF + отдельный фронт React/Next.js с самого начала

2. Аутентификация
- Аутентификация - только email/пароль, пользовательские сессии

3. Роли для MVP
- Минимальные набор ролей пользователей

4. Турниры, Тублица, Участники и Жеребьёвка — детали
- Ручная правка судьёй допускается и должна быть логирована.
- Очки за bye по умолчанию указываются в настройках турнира, по-умолчания - 1 очко
- Если нечётное число игроков в туре 1, то bye получает последний по посеву
- Если игрок выбывает (Withdrawn) по ходу турнира, он остается в таблице, однако больше не учавствует в составлении пар и не набирает очков (в т.ч. bye)
- В MVP таблица сортируется только по очкам, а при равенстве — по посеву (seed)
- Статусы турнира: переходы Draft → Registration → Running — ручные (кнопки судьи)
- Поле rating — опциональное? Если не задан — посев по порядку добавления

5. Публичность
- Публичные страницы — доступны полностью без логина
- Slug генерируется автоматически
- UI MVP на EN. Однако держим в голове, что языков может быть много
